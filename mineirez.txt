# MINEIREZ COMPILER – DOCUMENTAÇÃO DO ANALISADOR LÉXICO

1. VISÃO GERAL

---

O analisador léxico (lexer) é a primeira etapa do compilador Mineirez. Sua função é ler o código-fonte como texto e transformá-lo em uma sequência de tokens.

Tokens são unidades léxicas com significado, como palavras-chave, identificadores, números e operadores.

Exemplo:

Entrada:
uai trem = 10

Saída:
<TK_UAI, uai>
<TK_ID, trem>
<TK_ATRIB, =>
<TK_NUM, 10>

---

2. O QUE O LEXER ACEITA

---

O analisador léxico do Mineirez reconhece os seguintes elementos:

2.1 Palavras-chave (reservadas)

São palavras com significado fixo na linguagem:

* uai    → declaração de variável
* se     → condicional
* entao  → início de bloco condicional
* fim    → fim de bloco
* fala   → saída (print)

Essas palavras NÃO podem ser usadas como nomes de variáveis.

2.2 Identificadores (variáveis)

Representam nomes definidos pelo usuário.

Regras:

* Devem começar com letra
* Podem conter letras, números e underscore (_)

Exemplos válidos:

* trem
* x
* valor1
* minha_variavel

Exemplos inválidos:

* 1valor
* @teste

2.3 Números

Apenas números inteiros positivos são suportados no momento.

Regras:

* Sequência de dígitos (0-9)

Exemplos:

* 10
* 200
* 5

2.4 Strings

Representam textos entre aspas duplas.

Regras:

* Devem começar e terminar com "
* Conteúdo interno é livre (exceto quebra de string)

Exemplos:

* "uai sô"
* "trem bão demais"

2.5 Operadores

Símbolos que representam operações:

* =   → atribuição
* > → maior que
* <   → menor que
* + → soma
* - → subtração

Observação:
O operador == pode ser implementado futuramente.

2.6 Espaços em branco

* Espaços, tabs e quebras de linha são ignorados
* Servem apenas para separar tokens

2.7 Fim de arquivo

* Representado pelo token TK_EOF
* Indica que não há mais código para analisar

2.8 Erros léxicos

Qualquer símbolo não reconhecido gera:

<TK_ERRO, símbolo>

---

3. COMO O ANALISADOR LÉXICO FUNCIONA

---

O funcionamento do lexer é baseado em leitura sequencial do código.

Ele utiliza um ponteiro (posição) que percorre o texto caractere por caractere.

3.1 Inicialização

A função:

iniciar_lexer(codigo_fonte)

* Recebe o código como string
* Inicializa a posição como 0

3.2 Função principal: proximo_token()

Essa função é chamada repetidamente e retorna o próximo token encontrado.

Fluxo:

1. Ignora espaços em branco
2. Verifica o tipo do caractere atual
3. Decide qual função de reconhecimento chamar

---

4. PROCESSO DE RECONHECIMENTO

---

4.1 Identificadores e palavras-chave

Se o caractere atual for uma letra:

* O lexer entra na função reconhecer_id()
* Continua lendo enquanto for:

  * letra
  * número
  * underscore

Depois:

* Compara o valor com palavras-chave
* Se for igual → retorna token de palavra-chave
* Senão → retorna TK_ID

4.2 Números

Se o caractere for um dígito:

* Entra em reconhecer_numero()
* Continua lendo enquanto for número
* Retorna TK_NUM

4.3 Strings

Se encontrar aspas ("):

* Entra em reconhecer_string()
* Lê tudo até encontrar outra "
* Retorna TK_STRING

4.4 Operadores

Se o caractere for símbolo:

* Usa switch-case para identificar:

  * =
  * >
  * <
  * *
  * *

Retorna o token correspondente

4.5 Erros

Se o caractere não for reconhecido:

* Retorna TK_ERRO
* Guarda o caractere inválido

4.6 Fim do código

Quando encontra '\0':

* Retorna TK_EOF

---

5. ESTRUTURA DO TOKEN

---

Cada token possui:

* tipo  → categoria (TK_ID, TK_NUM, etc.)
* valor → conteúdo textual

Exemplo:

<TK_ID, trem>
<TK_NUM, 10>

---

6. EXEMPLO COMPLETO

---

Código:

uai trem = 10
fala "uai sô"

Saída:

<TK_UAI, uai>
<TK_ID, trem>
<TK_ATRIB, =>
<TK_NUM, 10>
<TK_FALA, fala>
<TK_STRING, uai sô>
<TK_EOF, EOF>

---

7. LIMITAÇÕES ATUAIS

---

* Não suporta números decimais
* Não suporta comentários
* Não suporta operador ==
* Não valida erros complexos de string
* Não diferencia tipos (int, string, etc.)

Essas melhorias podem ser implementadas nas próximas etapas.

---

8. CONCLUSÃO

---

O analisador léxico do Mineirez:

* Lê o código fonte
* Identifica padrões
* Gera tokens organizados

Ele é a base para as próximas etapas do compilador, como:

* Análise sintática (parser)
* Análise semântica
* Execução ou geração de código
